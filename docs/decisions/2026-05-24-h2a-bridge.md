# h2a host bridge schema

Date: 2026-05-24

Status: accepted

Mirrors: `@sentropic/h2a` DEC-059 (h2a side).

## Decision

Sentropic Remote ships a host-side JSON Schema, `h2aBridgeProfileSchema`
(`packages/protocol/src/schemas/h2a-bridge.ts`), that formalizes the contract
between a Sentropic Remote session Pod (the host) and an `@sentropic/h2a` MCP
sidecar running inside it (`h2a mcp-serve`).

The only V1 profile is the `remote` host itself, exported as the validated
constant `H2A_BRIDGE_PROFILE_V1`.

## Rationale

`@sentropic/h2a@0.1.22` (DEC-058 + DEC-059) renders a Kubernetes sidecar
manifest plus a formal contract describing what the sidecar expects from the
host (instance identity env vars, lifecycle state mapping, resource-limit
reflection, workspace disclosure boundary, auth boundary). That contract lived
only in h2a's `docs/k8s-sidecar.md`. Adding the mirror schema here makes the
contract enforceable symmetrically: the host can validate the profile it
advertises, and h2a can validate what it receives.

## Contract clauses (V1)

1. **identity** — instance template `remote:${SESSION_ID}`, env var map
   `H2A_INSTANCE` / `H2A_HOST` / `H2A_ROOT`, host hint `remote`.
2. **lifecycle** — maps Remote session states (`provisioning`, `running`,
   `terminating`, `ended`) onto the h2a vocabulary (`opening`, `live`,
   `draining`, `closed`, `expired`).
3. **resourceLimits** — the host reflects limits to the sidecar but does not
   let the sidecar enforce them (`enforced: false`).
4. **disclosure** — workspace boundary `/workspace`; cross-workspace access is
   `deferred` in V1.
5. **authBoundary** — in-Pod loopback transport (`mcp-serve` on localhost),
   enforced by the session Pod network policy.

## Adoption

- Schema added under `packages/protocol/src/schemas/`, re-exported from the
  protocol barrel; `H2AHostBridgeProfile` type inferred in `types.ts`.
- `H2A_BRIDGE_PROFILE_V1` is a pure constant in `packages/protocol` (no
  orchestrator dependency) so both protocol consumers and the
  `k8s-orchestrator` read the same source of truth when wiring the sidecar.
- Validation lives in `schema-validation.test.ts`.

## Versioning

- Any future change to the bridge contract requires a paired DEC on both
  sides (`@sentropic/h2a` and this repo) plus a minor bump.

## Note on naming

The h2a draft referenced the legacy repo codename `remote-controle`. This repo
was renamed to `rhanka/remote` on 2026-05-24, and the schema uses `remote` as
the canonical `hostId` / `hostHint` / instance prefix accordingly.
